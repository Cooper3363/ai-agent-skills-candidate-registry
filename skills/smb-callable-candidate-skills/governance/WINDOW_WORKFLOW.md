# 五窗口任务逻辑

## 研究窗口

目标：扩大候选面，但只围绕中小微企业工具。

输出：

- 业务包、部门角色、工作内容、候选 ID、来源链接。
- 是否已有 13 个稳交付 Skill 覆盖。
- 建议状态：`market_lead` 或 `needs_license_review`。
- 每轮优先输出每个业务包 Top 3，避免一次性把全部线索送测。

不做：

- 不直接宣布客户可调用。
- 不把来源项目当作验收结论。
- 不把通用开发工具、研究 demo、WebUI 产品优先送入候选库。

## 许可证窗口

目标：把候选从“看起来能用”拆成“能否合规试跑”。

输出：

- LICENSE、SPDX、README 商用限制、维护状态。
- 依赖/API/模型/OAuth/云服务风险。
- 试用边界标签：`mock_only`、`dry_run_only`、`BYOK_required`、`read_only`、`external_action_blocked`。
- 结论：可送 L2、需补资料、暂缓、只作线索。

不做：

- 不重复复核已完成结论，除非候选拆到新的真实上游。
- 许可证不清不得送正式 L2。
- 高风险动作不得放行真实执行。

## 测试台窗口

目标：分清候选 smoke 和正式 L2。

输出：

- 候选 smoke test：判断是否能进入候选调用库继续试跑。
- 正式 L2 simulated：每个候选至少 3 个中文业务用例。
- 结论只能是：L2 通过可封装、需补测、暂缓、未通过、仅组件。

不做：

- 不安装依赖。
- 不访问真实账号。
- 不真实发送、写 CRM、写日历、抓网页、生成图片或视频。
- 不把 simulated L2 写成真实 Harness 通过。

## 封装窗口

目标：把通过 L2 的候选变成可交接草案。

输出：

- 候选库封装：`CANDIDATE.md`、`candidate.yaml`、调用样例、限制说明。
- L3 封装：`SKILL.md`、`skill.yaml`、输入输出、权限边界、3 个中文测试用例、平台交接备注。

不做：

- 不封装未通过正式 L2 的候选为独立 Skill。
- 不把组件项写成独立 L3。
- 不把 draft L3 放入稳交付库。

## 平台调用验收窗口

目标：守住客户可调用门槛，同时允许候选库更快增长。

输出：

- 候选调用验收：是否可内部/受控试跑。
- 稳交付验收：是否可客户正式调用。
- 对每个 L3 检查 Skill ID、intent、adapter target、schema、权限、依赖、失败模式、人工复核触发和最小调用样例。

不做：

- 不验收只有文本摘要、没有 `SKILL.md` / `skill.yaml` 的候选。
- 不验收 `component_only` 为客户可调用 Skill。
- 不把候选调用通过等同于稳交付通过。
